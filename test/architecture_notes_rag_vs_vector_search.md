Architecture Notes: RAG vs. Vector Search
Clarify that these are complementary, not competing: Vector Search is a retrieval mechanism; RAG is an application architecture pattern that uses vector search as one of its components. Vector Search alone: use when you need semantic similarity (search, recommendations, deduplication) without LLM generation. RAG: use when you need LLM-generated answers grounded in your private data. Architecture comparison table: components, latency profile, cost drivers, failure modes. Deep dive on RAG with MongoDB Atlas: chunking strategy (fixed-size vs. semantic), embedding model choice (trade-offs: quality vs. cost vs. latency), $vectorSearch for retrieval, context window management, prompt construction. Hybrid RAG pattern: combine $vectorSearch with $search (full-text BM25) using Reciprocal Rank Fusion for better recall. When NOT to use RAG: small static datasets (fine-tuning may be better), real-time data (RAG has retrieval latency), highly structured queries (plain MongoDB queries are more reliable).
